# Digital FTE Factory — Project Constitution

Version: 1.0.0 | Ratified: 2026-02-18 | Last Amended: 2026-02-18

## Core Principles

### I. Code Quality Laws

ALL code MUST adhere to the following quality standards:

- **Type Hints**: ALL Python functions must have type hints for inputs and return values
- **Error Handling**: ALL async functions must have try/except with structured error logging
- **Secrets Management**: NO hardcoded secrets — use environment variables via pydantic-settings
- **Database Security**: ALL database queries must use parameterized statements (no f-strings in SQL)
- **API Response Format**: ALL API endpoints must return structured JSON with `{data, error, metadata}` envelope

**Rationale**: Consistent code quality reduces cognitive load, prevents security vulnerabilities, and enables better tooling support.

### II. AI Agent Laws

EVERY AI agent MUST follow these operational rules:

- **Tool Documentation**: EVERY agent tool must have a docstring describing when the LLM should call it
- **Input Validation**: EVERY tool must validate inputs via Pydantic before execution
- **Response Protocol**: Agents NEVER send a response without calling send_response tool
- **Ticket Creation**: Agents ALWAYS create a ticket before any other action
- **Channel Identity**: Channel identity (email/whatsapp/web_form) must be preserved in every DB write

**Rationale**: Agents operate autonomously; clear contracts and validation prevent cascading failures.

### III. Multi-Channel Laws

ALL multi-channel communications MUST follow these rules:

- **Customer Identity**: Customer identity is ALWAYS resolved by email (primary) or phone (secondary)
- **Response Length Limits**: Response length limits are HARD constraints:
  - Email: 500 words maximum
  - WhatsApp: 300 characters maximum
  - Web: 300 words maximum
- **Channel Formatting**: Channel-specific formatting is applied in formatters.py BEFORE send_response is called
- **Cross-Channel History**: Cross-channel history is ALWAYS loaded before generating any response

**Rationale**: Consistent customer experience across channels requires unified identity and context.

### IV. Database Laws

ALL database operations MUST follow these standards:

- **Timestamps**: ALL tables must have `created_at` with timezone
- **UUID Generation**: ALL UUIDs generated by `gen_random_uuid()` at DB level
- **Migrations**: Migrations tracked in `database/migrations/` — NEVER edit schema.sql directly in prod
- **Vector Search**: pgvector extension required for knowledge_base semantic search

**Rationale**: Database consistency ensures data integrity, auditability, and enables advanced features like semantic search.

### V. Testing Laws

ALL code MUST be covered by the following testing requirements:

- **Unit Tests**: Unit tests required for all tools and channel handlers
- **E2E Tests**: E2E tests required for all three channel flows (email, whatsapp, web_form)
- **Merge Gate**: No PR merges without passing transition tests
- **Performance Baseline**: Load test baseline: P95 < 3s, uptime > 99.9%

**Rationale**: Comprehensive testing prevents regressions and ensures system reliability under load.

### VI. Frontend Laws

ALL frontend code MUST follow these standards:

- **Validation**: All form inputs validated client-side AND server-side
- **Async States**: Loading/error/success states required on every async UI action
- **Component Library**: Shadcn components preferred over custom HTML
- **Styling**: No inline styles — use Tailwind utility classes only

**Rationale**: Consistent UX patterns and validation prevent user errors and improve maintainability.

### VII. Commit Message Laws

ALL commits MUST follow this format:

- **Format**: `<phase>(<scope>): <what changed>`
- **Scopes**: constitution, spec, clarify, plan, task, impl, db, agent, channel, k8s, test, ui
- **Example**: `impl(agent): add escalation tool with sentiment trigger`

**Rationale**: Structured commit messages enable automated changelogs and faster code review.

## Development Workflow

### Code Review Requirements

- ALL PRs MUST be reviewed by at least one team member
- Reviewers MUST verify constitution compliance
- ALL merge conflicts must be resolved before review approval

### Testing Gates

- ALL new features MUST include tests before merge
- CI MUST pass all tests before deployment
- Performance regression >10% requires team approval

### Deployment Process

- ALL deployments require passing CI/CD pipeline
- Feature flags MUST be used for incomplete features
- Rollback plan required for major releases

## Governance

This constitution supersedes all other development practices within this project.

### Amendment Process

Constitution amendments require:
1. Proposal with clear rationale
2. Team discussion and approval
3. Documentation of changes
4. Migration plan for existing code

### Versioning Policy

- **MAJOR**: Backward incompatible principle removals or redefinitions
- **MINOR**: New principle/section added or materially expanded guidance
- **PATCH**: Clarifications, wording, typo fixes, non-semantic refinements

### Compliance Review

- ALL PRs MUST verify constitution compliance
- Complexity deviations MUST be justified in PR description
- Quarterly constitution review recommended for continuous improvement
